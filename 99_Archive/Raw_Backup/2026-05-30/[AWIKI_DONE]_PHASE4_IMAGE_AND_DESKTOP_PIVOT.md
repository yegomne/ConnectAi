## 2. 모델 선정 여정 (검증으로 골라냄)

### 2-1. NSFW 검수 모델

| 모델 | 결과 | 비고 |
|------|------|------|
| llama3.2-vision | ❌ 폐기 | 5개 검증이미지 전부 🟡 오판, 음부 완전노출(C-913)도 "노출없음" 판정(치명적), VRAM 13.1GB로 qwen3과 동시사용 불가, 13초 느림 |
| **NudeNet** | ✅ 채택 | 0.02초, VRAM 거의 0, qwen3과 공존 가능. 실제 사람 노출(D-415 유방 0.85) 정확 검출 |

**NudeNet의 약점**: 실리콘 제품(모형) 노출은 못 잡음.
- C-913(실리콘 핸드잡, 음부 노출)을 검출 0건, threshold 0으로도 안 잡힘.

**약점 보완 = 카테고리 위험도 2중 안전망**:
- HIGH(리얼돌/바디/먹쇠/실리콘 등) = 무조건 🟡 이상
- MID(명기/핸드잡/란제리/애널 등) = NudeNet 🟢라도 🟡로 올림
- LOW(홀컵/진동기/콘돔/젤/SM 등) = NudeNet 신뢰
- 기본값 MID. `CATEGORY_RISK` 딕셔너리(사장님 수정 가능)
- **검증 성공**: C-913이 핸드잡/리얼돌 카테고리에서 🟡로 올라감, C-732(홀컵 LOW)는 🟢 유지

### 2-2. 누끼(배경 제거) 모델

| 모델 | 품질 | 속도(CPU) | 결과 |
|------|------|-----------|------|
| u2net | 흰색 라벨 잘림 | ~1초 | ❌ 흰색 약점 |
| isnet-general-use | 최악 | ~1.7초 | ❌ 사용 불가 |
| **birefnet-general** | 최고 | 9~20초 | ✅ 채택 |

- **검증 (C-732 홀컵)**: u2net은 흰색 라벨/입구를 배경으로 오인해 잘림.
  BiRefNet은 검정 제품 경계 + 흰색 라벨 + 흰색 입구까지 완벽 보존.
- **검증 (K-552 글랜스 트레이너)**: 투명/반투명 부품도 BiRefNet 잘 처리됨.
- BiRefNet은 품질 최고지만 CPU에서 본질적으로 느림(이미지 크기 무관 9~20초).

### 2-3. CLIP 분류기 = 옵션 2 (나중)

지금은 **NudeNet + 카테고리(2중 안전망)** 로 가되, 사장님이 검수한 이미지를
학습 폴더에 자동 축적 → 나중에 데이터 30장+ 쌓이면 CLIP 학습 → 3중 안전망.
- `training_collector.py`: explicit/safe/caution 폴더 + `_labels.jsonl`
- 핵심 = `corrected` 플래그: NudeNet 판정과 사장님 판정이 다를 때 = 가장 가치있는 학습데이터
- **검증 성공**: C-913 → NudeNet "caution", 사장님 "explicit" → corrected=True ⭐

## 3. GPU 시도와 무산 (중요한 교훈)

### 3-1. 원래 계획
누끼/스크리너를 세컨 PC FastAPI로 옮겨 RTX 5060 Ti GPU로 가속 (1~2초 기대).

### 3-2. 진행
- gui 5개 모듈을 메인 PC → 세컨 PC(Z:\)로 복사
- `src/server.py`에 `/screen_image`, `/cutout` 엔드포인트 추가
- `/health`에 `image_models`(onnx_providers) 추가
- onnxruntime-gpu 설치, GPU 인식 확인(`CUDAExecutionProvider` 목록에 보임)

### 3-3. 무산 — CUDA 버전 불일치
```
세컨 PC: CUDA 13.2 설치됨
onnxruntime-gpu 1.26.0: CUDA 12 DLL(cublasLt64_12.dll) 요구
  → DLL 없음 → GPU 로딩 실패 → CPU 폴백
  → 큰 이미지를 CPU로 처리하다 135초 + 메모리 폭발(float64, bad allocation)
  → 파섹(원격제어) 다운
```
- `pip install nvidia-cublas-cu12 nvidia-cudnn-cu12` 설치해도 onnxruntime이
  DLL 경로(site-packages\nvidia\...)를 못 찾아 실패.

### 3-4. 결정 — 이미지 처리는 메인 PC 로컬 CPU로 (D-031, 이후 재논의)
- **사장님 지적이 결정적**: "CPU로 할 거면 메인 PC(Ryzen 5 7600)가 세컨 PC(4650G)보다 빠른데 왜 세컨에서?"
- 메인 7600(Zen4) > 세컨 4650G(Zen2)이므로 CPU 누끼라면 메인 PC가 맞음.
- 역할 분담:
  - 세컨 PC = qwen3(상품명) 전용 GPU
  - 메인 PC = 누끼 + 스크리너 + 축적 + GUI (CPU)
- 세컨 PC에 옮긴 누끼 코드는 방치(나중 GPU 풀리면 살림).

### 3-5. 메모리 최적화 (CPU 안정화)
- `make_cutout()`에 입력 리사이즈 추가: `MAX_CUTOUT_INPUT = 1500` (긴 변 기준 축소)
  → 큰 이미지(상세페이지 부분 등) 메모리/속도 개선. 썸네일이 1000px라 원본 불필요.
- 검증 이미지가 다 1500px 이하라 리사이즈 효과는 안 보였지만 코드는 정상.
- BiRefNet CPU 속도: 9~15초 (단건이라 수용 가능, 사장님 판단).

## 4. 최종 워크플로우 (사장님이 정리, 여러 번 시각화로 확정)

### 4-1. 큰 그림
```
1. 상품명을 넣는다 → 상품명 제작  (즉석 완성: 5+1 후보 선택)
2. 이미지를 넣는다 → 이미지 제작  (누끼 + 썸네일 + NSFW)
```

### 4-2. 상품명 = 즉석 완성 (입력 단계에서 바로)
- qwen3 8초로 5+1 후보 생성 → 사장님이 즉석 선택.
- 이유(사장님): "상품명은 입력 때 완성이 완벽." 빠르고 판단이 필요하니 즉석.

### 4-3. 이미지 처리 올바른 순서 (사장님 지적으로 바로잡음)
```
입력 (기존 썸네일 OR 상세페이지)
  ↓
① 자동 NSFW 검수 (먼저! 노출 위치 파악)   ← NudeNet 0.02초
  ↓
② 사장님 편집 (마우스 드래그)
   - 노출 부위 → 흰 박스 드래그로 가리기
   - 썸네일 만들 영역 → 드래그 → 누끼 + 1000×1000
  ↓
③ 검수 결과 = 학습 데이터 축적 (corrected 시 CLIP용)
  ↓
완성
```
- **이전 오류**: 가리기를 편집·검수 양쪽에 중복으로 넣고, 편집을 검수보다 먼저 그림.
- **바로잡음**: 검수 먼저(노출 위치 파악) → 편집(가리기+썸네일) 나중. 가리기는 편집에 1회.

### 4-4. 썸네일 두 가지 소스 (사장님 원래 구상)
- ① 기존 썸네일 이미지로 작업 가능
- ② **상세페이지에서 부분 드래그**해서 생성 가능 ← 아이템위너 회피 핵심
- 둘 다 누끼 + 1000×1000 흰배경 썸네일.

### 4-5. 상세페이지도 1급 시민 (제가 처음에 놓쳤던 것)
- 처음 4-2 설계: "긴 상세페이지는 누끼 제외"
- 실제: 상세페이지는 **누끼 재료 + NSFW 가리기 대상** 둘 다.
- "통째로 누끼"만 제외, "부분 잘라 누끼"는 핵심 기능.
- 상세페이지(최대 30000px)는 화면에 축소 표시 + 드래그 영역의 원본 좌표 환산 필요.

### 4-6. 처리 방식 = C (가장 단순, 한 건씩 완성) ★ 채택 (D-034)
- 한 아이템을 처음부터 끝까지 완성 후 다음.
- 일괄(B안: 지정/실행 분리)은 복잡도 높아 나중에. "써보고 추가."
- 5~10개 아이템 묶음 작업 상정.
- (B안 = 드래그로 좌표만 기록 → [한번에 작업]으로 누끼 일괄 실행.
  검수는 빠르니 입력 때, 누끼만 일괄. 가능하지만 복잡 → 보류.)

## 5. 🔀 핵심 전환: 웹(Streamlit) → 데스크탑 앱

### 5-1. 왜 전환하나 (관제탑 핵심 기술 = 마우스 드래그 영역 선택)

관제탑의 심장은 **이미지 위에서 마우스 드래그로 영역을 고르는 것**:
- 썸네일 만들 영역 드래그
- 노출 가릴 곳 드래그 (흰 박스)

#### Streamlit으로 시도한 것 (실패)
- `streamlit-drawable-canvas 0.9.3` (2023년, 미유지보수) 검증.
- streamlit 1.58.0과 격차가 커서 Claude Code가 shim(임시 호환코드)으로 억지 작동.
- **검증 결과**:
  - ✅ 드래그로 사각형 그려짐(14개), ✅ 좌표 추출 + 원본 좌표 환산 정상, ✅ 흰 박스 모드 됨
  - ❌ **치명적: 배경 이미지(`background_image`)가 캔버스에 안 보임** (빨간 사각형만, 제품 이미지 안 깔림)
  - 사장님: "이미지 미리보기가 안돼서 정확하게 못함"
- 원인: Fabric.js 기반 라이브러리가 최신 streamlit과 안 맞아 배경 표시 깨짐.
- 버전 맞추기 불가: drawable-canvas 0.9.3은 streamlit 1.2x대 필요 →
  streamlit 낮추면 3단계 GUI(st.dataframe on_select 등 1.35+ 기능)가 깨짐.

#### HTML 캔버스도 검토했으나
- 순수 HTML/JS `<canvas>` + components.html()로 가능은 함(이미지 base64 직접 그리기).
- 하지만 사장님이 **처음부터 제기**: "데스크탑 프로그램으로 돌리면?"

### 5-2. 데스크탑이 맞는 이유
- **이미지 편집(드래그, 영역선택, 가리기)은 데스크탑이 본질적으로 강함** (포토샵이 웹 아닌 이유).
- 웹은 마우스 정밀 편집이 약하고 라이브러리 의존으로 깨짐.
- 데스크탑은 네이티브 캔버스 → 이미지 표시 확실, 드래그 자연스러움, 라이브러리 씨름 없음.

### 5-3. 데스크탑 + 세컨 PC 실행 = GPU 부활 가능 ⭐ (사장님 통찰)

사장님 질문: "데스크탑으로 가면 아까 실패했던 세컨 컴 GPU 사용 다시 시도 가능해?"

**답: 가능하다. 그리고 데스크탑이면 GPU 활용이 더 유리해진다.**

```
[현재 웹 구조]
메인 PC(웹 GUI, AMD) ←네트워크(FastAPI)→ 세컨 PC(qwen3 GPU)
누끼는 메인 CPU (AMD라 GPU 불가)

[데스크탑 앱을 세컨 PC에서 실행] ⭐
세컨 PC(RTX 5060 Ti)에서 데스크탑 앱 직접 실행
  - 상품명 qwen3: 로컬 GPU
  - 누끼 BiRefNet: 로컬 GPU (네트워크 없음)
  - 이미지 편집: 데스크탑 네이티브 (드래그 강력)
메인 PC는 파섹으로 화면만 봄
  = 모든 게 한 PC(세컨)에서 GPU로, 편집도 네이티브
```

- 사장님은 이미 **파섹으로 세컨 PC를 원격제어** 중 → 세컨 PC에 앱 깔고 파섹으로 쓰면
  GPU 직접 활용 + 이미지 편집 둘 다 해결.
- **단, CUDA 13 vs 12 문제는 여전히 풀어야 함** (데스크탑이라고 자동 해결 X).
  - 아직 안 해본 방법: onnxruntime-gpu 버전 맞추기 / **PyTorch 기반 BiRefNet**(CUDA 유연) /
    DLL 경로 환경변수 명시.
  - PyTorch BiRefNet이 유망: Ollama처럼 CUDA를 더 유연하게 관리.

## 6. 데스크탑 전환 — 아키텍처 선택지 (다음 세션에서 결정)

### 6-1. GUI 구조
- **구조 A**: 전부 데스크탑 앱으로 통합 (상품명+이미지 한 프로그램).
  - 백엔드(qwen3 호출, 검증, 강화학습, 누끼, 검수)는 기존 코드 재활용, UI만 새로.
  - 장점: 워크플로우 매끄러움. 단점: 3단계 Streamlit UI 재작성 필요.
- **구조 B**: 상품명 웹 유지 + 이미지만 데스크탑 (분리). → 왔다갔다 불편, 비권장.
- **구조 C**: 데스크탑 통합 + 상품명 백엔드 로직 재사용 (A와 거의 같음).

**제안: A 또는 C** (데스크탑 통합, 백엔드 재활용).

### 6-2. 데스크탑 GUI 도구
- **PySide6 / PyQt6**: 가장 강력, 이미지 캔버스/드래그/영역선택 전문적. 코드량 많음. ★ 권장
- **Tkinter**: 파이썬 기본 내장, 가볍지만 투박.
- **CustomTkinter**: Tkinter 모던 버전, 중간.
