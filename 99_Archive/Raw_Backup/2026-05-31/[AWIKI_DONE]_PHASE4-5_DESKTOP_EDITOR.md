- 검증: C-907 상단에 흰 박스 → 원본 좌표 (0,0) 크기 860×158, "좌상우 경계까지 덮음" ✅

## 4. 두 박스 + 광고 가리기 (모드 3종)

### 박스 용도 정리 (사장님 헷갈려서 재확인)
- 🟥 **빨간 박스 (영역 선택)**: 이 부분을 잘라서 썸네일 만들기 (누끼 대상). 아이템위너 회피(B).
- ⬜ **흰 박스 (노출 가리기)**: 노출 부위 흰색으로 덮기. 마켓 반려 방지(A). **학습됨**.
- 🟦 **파랑 반투명 (광고 가리기)**: 리보스 광고/로고/사이트 안내 덮기. **학습 안 함**.

### 왜 광고 가리기를 분리했나 (중요)
사장님 지적: 맨 위/아래에 리보스 사이트 광고가 있어 가려야 하는데,
이걸 "노출"로 학습하면 안 됨 (CLIP이 "REBO 로고 = 노출"로 잘못 배움).
→ 가리기를 노출(학습O) / 광고(학습X)로 구분.

모드 3종 확정:
- 영역 선택: 빨강 반투명 [Shift+S] — 썸네일용
- 노출 가리기: 흰 박스 불투명 [Shift+D] — 학습됨 ⭐
- 광고 가리기: 파랑 반투명(전체 채움) [Shift+A] — 학습 안 함

저장 시: 노출·광고 둘 다 **흰색**으로 합성 (화면은 파랑이지만 출력은 흰색).

## 6. 4-5b-2 — 가리기 합성 + 학습 축적 (안전 로직)

### 핵심: 학습 데이터 축적 방식 = "가리면 자동"
- 노출 가리기(흰 박스)를 그리면 = "여기 노출이다" 판정 → explicit
- 사장님 평소 작업(가리기)만으로 학습 데이터 쌓임 (3단계 키워드 강화학습과 동일 패턴)

### ★ 학습 안전 로직 (버그 발견 → 수정, 매우 중요)
**발견된 버그**: D-415(명백한 누드)에 노출 가리기 안 했더니 "safe (corrected)"로
축적됨 → 누드가 "안전"으로 잘못 학습될 뻔. 학습이 독이 되는 위험.

**수정된 3-way 로직** (N = 노출가리기 박스 개수):
```
CASE 1) N >= 1 (사장님이 노출 가림):
   → label = "explicit"
   → corrected = (nudenet_level != "explicit")  # NudeNet 놓쳤으면 corrected ⭐
   → 축적 O

CASE 2) N == 0 AND nudenet == "safe" (둘 다 노출 없다 동의):
   → label = "safe", 축적 O

CASE 3) N == 0 AND nudenet in (caution, explicit) (모순!):
   → "safe로 학습" 절대 금지 (누드를 안전으로 학습하면 재앙)
   → 학습 보류 + 경고 표시
   → 'safe 확정' 버튼으로만 의도적 safe 축적 가능
```

검증 (둘 다 정확):
- 누드에 노출 가리기 함 → explicit 축적 (NudeNet도 explicit, 일치) ✅
- 누드에 안 가림 → ⚠️ 학습 보류 + 경고 (safe로 안 들어감) ✅
- **실리콘(NudeNet caution)에 노출 가리기 → explicit corrected ⭐** 실제 확인됨!
  → NudeNet 약점을 사장님 작업으로 메우는 데이터가 실제로 쌓이기 시작.

### 광고 가리기는 학습 제외
- 광고 가리기(파랑) 박스는 label 판정에서 완전 제외 (학습 오염 방지)

### 합성 저장
- 노출+광고 박스 → 흰색 합성 → `data/processed/masked/` (원본명_masked.확장자)
- 재활용: training_collector.collect_sample(image_path, label, category_key, nudenet_level)

## 7. 4-5b-3 — 누끼 → 썸네일 (+ 누끼 사용/미사용)

### 처리 (가리기 적용된 이미지 기준 — D 결정 A)
- 빨간 영역 → 가려진 이미지에서 crop → 누끼 → 1000×1000 썸네일
- "가려진 이미지에서" 누끼 = 썸네일에도 노출 가려진 상태 (안전)
- 재활용: image_processor.make_cutout(BiRefNet), make_thumbnail(1000 흰배경 가운데정렬)
- 저장: cutout PNG (`data/processed/cutout/`), thumbnail JPG (`data/processed/thumbnail/`)

### 누끼 사용/미사용 토글 (사장님 지적)
- 모든 영역이 누끼 필요한 건 아님. 이미 깔끔한 디자인 컷은 자르기만.
- [누끼 사용]: crop → BiRefNet 누끼 → 썸네일 (배경 제거, 9~15초 CPU)
- [누끼 미사용]: crop → 누끼 생략, 바로 1000×1000 흰배경 가운데 배치 (즉시, 빠름)

검증: 인물 부분 빨간영역 → 누끼 → 흰배경 썸네일 정상. corrected ⭐도 확인.

### 화면 표시 vs 저장 품질 (사장님 의문 해소)
- 화면 = 축소 미리보기 (긴 이미지 0.789배 등)
- 저장 = 원본 좌표 기준 1000×1000 (품질 유지)
- 즉 화면이 작아 보여도 실제 저장은 원본 해상도. 잘못 본 게 아님.
- 단 MAX_CUTOUT_INPUT=1500: 누끼 전 1500px로 축소(메모리/속도). 최종 1000이라 무난.

## 8. 4-5 레이아웃 3단 + 줌 (협소 문제 해결)

### 문제
세로로 쌓인 구조(상단버튼/중간이미지/하단데이터) → 이미지 표시 협소.

### 해결: 3단 레이아웃 (QSplitter)
```
┌──────┬────────────────────┬──────────────┐
│ 좌측  │  중앙 캔버스         │  우측 패널    │
│ 목록  │  (가장 넓게)         │  검수/좌표    │
│      │  줌/스크롤          │  (접기 가능)  │
└──────┴────────────────────┴──────────────┘
상단 툴바: 모드 버튼 + 액션 버튼
```
- 우측 패널 [◀접기/펴기] → 긴 이미지 볼 때 캔버스 최대화
- **줌**: 마우스 휠(Ctrl+휠) + 버튼([+][-][100%][화면맞춤]) + 줌% 표시
- 줌 핵심: QGraphicsView mapToScene으로 줌/스크롤 무관 원본좌표 정확 (clamp 유지)
- 긴 이미지: Fit(전체) ↔ 줌인(정밀) 전환

## 9. 4-5c-1 — 제품 단위 그룹핑 + 세로 합치기 (사장님 핵심 지적)

### 배경 (워크플로우를 바꾼 통찰)
실제 마켓 등록 단위 = **제품 1개 = 여러 파일**.
공급사가 상세페이지를 분할해서 줌: ex001.jpg, ex002.jpg, ex004.jpg = 한 제품.
→ "이미지 1장 = 1작업"이 아니라 "제품 1개(여러 파일) = 1작업".

### 파일명 숫자로 그룹 자동 인식
- 파일명 끝 숫자 떼고 접두어가 같으면 한 그룹.
  정규식 `^(.*?)(\d+)$` → 접두어 + 숫자. ex001/002/004 → "ex" 그룹.
- 좌측 목록을 그룹으로 묶어 표시 (그룹 헤더 + 파일 + ✅ 표시)

### 결정 사항
- Q1 너비 다르면: **가장 넓은 것 기준**, 좁은 건 가운데 정렬(흰 여백)
- Q2 결번(003 없음): **있는 파일만 숫자 순서대로** (001→002→004)
- Q3 합치기 시그널: 파일별 완료(✅) + [제품 합치기] 버튼 (A+B 조합)
- Q3-2 미완료 파일: **저장(완료)한 것만 합침** (체크 안 한 건 제외)
- Q4 썸네일: 파일당 0~1개 생성 가능, **제품당 총 1개 이상**이면 OK

### ★ 저장 = 완료 자동 연동 (사장님 정리)
- 별도 완료 체크 불필요. "가리기 적용·저장" 누르면 = 그 파일 완료(✅) 자동.
- 저장한 파일만 합치기 대상. 안 한 파일은 제외.
- 그룹 헤더에 진행 표시 (예: "C-914 (4/4 저장됨)")

### 세로 합치기
- 저장된 파일만, 숫자 순서대로, 편집본(masked) 사용
- 가장 넓은 너비 기준, 좁은 건 가운데 정렬
- 저장: `data/processed/merged/` (그룹접두어_merged.jpg)

검증: C-914 그룹 4/4 → 세로 9551px로 합쳐짐 (편집본 4 + 원본 0) ✅
