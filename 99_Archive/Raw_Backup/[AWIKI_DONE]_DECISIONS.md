**환경 제약**: RTX 5060 Ti 16GB VRAM

**고려한 후보**

- B) qwen3.6:27b (17GB) — 보유 중
- C) qwen3.6:35b-a3b (23GB) — 보유 중, MoE
- D) qwen3:14b (9GB) — 신규 다운로드
- E) exaone3.5:7.8b (4.8GB) — LG 한국어 특화

**선택**: **D) qwen3:14b**

**근거**

- E는 한국어 자연스러움은 강하지만 명령어 추종이 14b 대비 약함
  - 보조 옵션으로 향후 비교 가능

**검증**
- 6개 카테고리 × 5후보 = 30개 후보 100% 통과

## D-003: thinking 모드 ON 유지

**상황**: qwen3 thinking 모드 제어 방식

**문제 발생**
- 초기 프롬프트에 `/no_think` 지시자 사용
- 14b로 격상 후 → **모든 응답이 빈 JSON `{}`** 으로 반환됨
- 0.4초 만에 끝나는 의심스러운 패턴 → 추론 자체를 안 함

**원인 분석**
- qwen3 14b는 thinking 의존도가 8b보다 큼
- `/no_think` + `format=json` 조합이 모델 내부에서 충돌
- 모델이 "생각도 안 하고 답도 안 하는" 상태로 빠짐

**해결책**
- 프롬프트에서 `/no_think` 제거
- Ollama API 페이로드에 `"think": false` 정식 옵션 사용 (현재 코드 적용)
- 빈 응답 시 자동 재시도 + thinking 켜고 폴백 메커니즘

**검증**: 14b + thinking 정식 옵션 사용 후 응답 시간 7~9초로 안정화, JSON 파싱 실패 0회

## D-006: 자동 교정 후처리 도입 (LLM 한계 보완)

**상황**: 프롬프트로 카테고리 격리 규칙 명시했으나 LLM이 무시

**구체 사례**: M-314 (애널/후방) — 프롬프트에 "명기 키워드 금지" 명시했지만 LLM이 5/5 모두 "명기" 포함

**원인**: LLM 프롬프트 따르기 능력의 본질적 한계 (특히 음/양성 키워드 제한)

**해결**: 코드 단에서 강제 치환
```python
CATEGORY_REPLACEMENT_MAP = {
    "후방": {"명기컵": "후방케어", "명기": "후방", ...},
    "여성": {"피스톤": "진동", "명기": "마사지기", ...},
    "팬티": {"진동": "디자인", "혼자 즐기는": "야간 데이트", ...},
    # ...
}
```

**처리 순서**
1. LLM 원본 생성
2. **카테고리 오염 자동 교정** ← 핵심 후처리
3. 단어 중복 제거 (max 2회)
4. 글자수 확인
5. 필요시 부스터 키워드 추가
6. 부스터 추가 후 다시 단어 중복 정리

**검증**: M-314 5/5 모두 "명기"→"후방" 자동 교정 성공

**철학**: LLM은 80% 잘 하고 후처리가 나머지 20% 강제 보장. 둘 다 필요.

## D-007: 단어 중복 제거 — max 2회

**상황**: 자동 교정 + 부스터 추가 후 같은 단어 3회 이상 등장하는 경우 발생

**예시 (수정 전)**
```
"후방 리얼터치 후방케어 강력진동 K 리어 자극기 10진동 후방 자극기"
                                                    ↑           ↑
                                              "후방" 3회, "자극기" 2회
```

**고려한 후보**
- A) 모든 중복 제거 (max 1회)
- B) max 2회까지 허용
- C) 중복 제거 안 함 (LLM 출력 그대로)

**선택**: **B) max 2회**

**근거**
- A: 쿠팡 SEO에서 핵심 키워드 2회 등장은 검색 가중치에 유리 → 손해
- C: 3회 이상이면 스팸 키워드 나열로 보일 위험
- B가 SEO 효과 유지하면서 자연스러움 보장

**검증**: 수정 후 30개 후보 중 단어 중복 문제 사라짐 (한 후보의 LLM 자체 오타 "쿠플용"만 잔존, 사장님 검수에서 거름)

## D-010: 부스터 키워드 카테고리별 분기

**상황**: LLM 응답이 35자 미만일 때 키워드 자동 추가

**문제**: 초기엔 모든 카테고리에 "어덜트굿즈"만 추가됨 → 5개 후보 끝부분 동일

**해결**
```python
BOOSTER_MALE = ["어덜트굿즈", "19금용품", "남성전용", "프리미엄", "신상"]
BOOSTER_FEMALE = ["여성전용", "이너케어", "릴렉서", "프리미엄", "신상"]
BOOSTER_ANAL = ["후방케어", "백도어용품", "19금용품", "프리미엄", "신상"]
BOOSTER_LINGERIE = ["이너웨어", "섹시속옷", "19금속옷", "프리미엄", "신상"]
BOOSTER_COUPLE = ["커플용품", "롤플레이", "코스튬", "프리미엄", "신상"]
```

**추가 규칙**
- 후보 인덱스(0~4)별로 순환 적용 → 5개 후보가 다른 부스터로 끝남
- 이미 LLM 원본에 해당 키워드 있으면 다음 키워드로 건너뛰기
- 부스터 키워드는 띄어쓰기 없는 단일 토큰 (`"19금용품"` ○, `"19 금용품"` ✗)
  - 이전 버그: 토큰 분할 후 재조합 시 "19 금용품"으로 깨짐 → 수정 완료

## D-013: 검증 단계의 다중 안전망

**상황**: 단일 검증으로는 모든 케이스 못 잡음 (검증된 사실)

**채택한 다중 안전망 구조**
