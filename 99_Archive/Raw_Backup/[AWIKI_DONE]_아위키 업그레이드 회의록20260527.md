# 🚀 A-Wiki (v2.0) 시스템 업그레이드 및 운용 매뉴얼 (최종)

## 📌 한 줄 통찰
> 아위키를 단순한 문서 요약기에서 벗어나, 문서를 가장 작은 단위로 해체(Atomic Split)하고 연결하며, '임시 작업장'과 '격리된 원본 백업소'를 통해 볼트의 무결성을 완벽하게 유지하는 '자율 지식 정원사'로 진화시켰다.

---

## 1. 🌟 주요 업그레이드 내용 (Core Features)

### ① 원자적 분해 (Atomic Split / Note Surgeon)
- **과거 (창고형):** 여러 주제가 섞인 긴 문서(예: 게임 플랫폼 기획안)를 통째로 하나의 파일로 저장하여 RAG 검색 시 노이즈(Hallucination) 유발.
- **업그레이드 (레고 블록형):** 긴 글을 독립적인 맥락을 가진 여러 개의 원자적 노트로 잘게 쪼개고, 쪼개진 노트 간에 위키링크(`[[ ]]`)를 촘촘히 연결. 필요한 지식 블록만 정확히 꺼내어 수익화 콘텐츠에 재사용 가능.

### ② '투트랙' 모니터링 (00_Raw & temp)
- 공식 수집함인 `00_Raw/` 폴더 외에, 넣고 빼기가 자유로운 샌드박스(Sandbox) 개념의 `temp/` 폴더를 신설하여 아위키가 두 곳 모두를 실시간 모니터링하도록 지침 수정.

### ③ 자동 아카이빙 (Auto-Archiving) 메커니즘
- `temp/` 폴더의 지저분한 파일 입출력 기록이 Git 커밋 히스토리를 오염시키는 것을 방지.
- 아위키가 `temp/`에서 문서 분해(지식화)를 완료하면, 원본 파일을 삭제하지 않고 즉시 `99_Archive/Raw_Backup/` 폴더로 이동(Move)시킴.
- **결과:** `temp/` 폴더는 항상 깨끗하게 비워지고, 원본 파일은 아카이브 폴더에 모여 깃허브(GitHub)에 안전하게 영구 백업됨.

### ④ 원본 백업소의 '격리 구역(Blacklist)' 설정
- 백업된 원본 문서가 추후 RAG 검색 결과에 중복으로 딸려 오거나, 리팩토링 과정에서 재차 수정되는 것을 방지.
- 아위키의 시스템 프롬프트(`skill.md`)에 `99_Archive/` 폴더를 **"접근 금지 구역"**으로 명시하여, 원본 데이터의 훼손 및 검색 오염을 원천 차단.

## 2. 📂 필수 폴더 구조 및 환경 설정 (Setup Guide)

아위키 v2.0이 에러 없이 완벽하게 구동되기 위해 볼트 최상단에 다음 구조를 반드시 세팅해야 함.

- `00_Raw/` : 보존이 필요한 공식 원시 데이터 수집용
- `10_Wiki/` : 아위키가 정제하고 쪼개놓은 지식 결과물이 저장되는 메인 공간
- `temp/` : **[신규]** 자유롭게 파일을 던져놓는 일회성 임시 작업장
- `99_Archive/Raw_Backup/` : **[신규]** 처리가 끝난 원본 파일이 자동 보관되는 백업소 (경로 오류 방지를 위해 최초 1회 수동 생성 필수)

### ⚠️ Git 무결성을 위한 핵심 설정 (`.gitignore`)
시스템 간의 명확한 역할 분담을 위해 볼트 최상단의 `.gitignore` 파일에 아래 내용을 정확히 반영해야 함.
- **`temp/` (추가 필수):** 지저분한 임시 작업 흔적을 깃허브에 남기지 않기 위해 Git 추적에서 철저히 배제함.
- **`99_Archive/` (절대 추가 금지):** 이 폴더는 깃허브에 원본을 '영구 백업'하는 것이 목적이므로, Git이 정상적으로 추적하여 동기화할 수 있도록 `.gitignore`에 넣으면 안 됨. (접근 차단은 오직 아위키의 `skill.md` 지침으로만 통제)

## 3. 🤖 아위키 작동 및 운용 방식 (Operation Modes)

아위키는 별도의 시간 주기로 스캔하는 것이 아니라, **이벤트 기반(Event-driven)**으로 파일이 생성되는 그 즉시(0초 딜레이) 반응함.

### Mode A: 자동 사냥 (기본 모드)
- **방법:** `00_Raw/` 또는 `temp/` 폴더에 마크다운 파일이나 문서를 넣는다. (채팅창에 명령할 필요 없음)
- **작동:** 세컨드 PC의 32GB 램 환경에서 돌아가는 로컬 서버가 파일 생성 이벤트를 즉각 감지하여 백그라운드에서 크롤링 ➔ 분해 ➔ 병합 ➔ 아카이빙 ➔ 커밋을 조용히 자동 수행함.

### Mode B: 수동 타겟팅 (특정 경로 지시)
- **방법:** 채팅창에 특정 파일의 경로를 짚어주며 직접 명령함. (예: "아위키, `10_Wiki/사업기획/플래쉬게임허브기획.md` 쪼개서 다시 작업해.")
- **작동:** 모니터링 구역을 벗어나, 지시받은 특정 경로로 즉시 이동하여 NLP 파이프라인을 가동함. 과거 문서를 재구조화(Refactoring)할 때 유용.

---

## 4. 💡 실무 적용 팁
- **과거 문서 리팩토링:** 기존의 무거운 파일들을 한 번에 다 다시 작업하라고 지시하면 서버에 과부하가 걸릴 수 있으므로, 폴더 단위로 쪼개어 점진적으로(Phased Rollout) 수동 타겟팅 명령을 내리는 것을 권장.
- **일상 업무:** 기획서나 참고 문서를 `temp/`에 "툭" 던져놓고 "작업해"라고만 지시하면 끝. 원본 백업부터 지식 구조화까지 아위키가 모두 알아서 처리함.
