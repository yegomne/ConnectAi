**검증**

## D-003: thinking 모드 ON 유지

**상황**: qwen3 thinking 모드 제어 방식

**문제 발생**
- 초기 프롬프트에 `/no_think` 지시자 사용
- 14b로 격상 후 → **모든 응답이 빈 JSON `{}`** 으로 반환됨
- 0.4초 만에 끝나는 의심스러운 패턴 → 추론 자체를 안 함

**원인 분석**
- qwen3 14b는 thinking 의존도가 8b보다 큼
- `/no_think` + `format=json` 조합이 모델 내부에서 충돌
- 모델이 "생각도 안 하고 답도 안 하는" 상태로 빠짐

**해결책**
- 프롬프트에서 `/no_think` 제거
- Ollama API 페이로드에 `"think": false` 정식 옵션 사용 (현재 코드 적용)
- 빈 응답 시 자동 재시도 + thinking 켜고 폴백 메커니즘

**검증**: 14b + thinking 정식 옵션 사용 후 응답 시간 7~9초로 안정화, JSON 파싱 실패 0회

## D-006: 자동 교정 후처리 도입 (LLM 한계 보완)

**상황**: 프롬프트로 카테고리 격리 규칙 명시했으나 LLM이 무시

**구체 사례**: M-314 (애널/후방) — 프롬프트에 "명기 키워드 금지" 명시했지만 LLM이 5/5 모두 "명기" 포함

**원인**: LLM 프롬프트 따르기 능력의 본질적 한계 (특히 음/양성 키워드 제한)

**해결**: 코드 단에서 강제 치환
```python
CATEGORY_REPLACEMENT_MAP = {
    "후방": {"명기컵": "후방케어", "명기": "후방", ...},
    "여성": {"피스톤": "진동", "명기": "마사지기", ...},
    "팬티": {"진동": "디자인", "혼자 즐기는": "야간 데이트", ...},
    # ...
}
```

**처리 순서**
1. LLM 원본 생성
2. **카테고리 오염 자동 교정** ← 핵심 후처리
3. 단어 중복 제거 (max 2회)
4. 글자수 확인
5. 필요시 부스터 키워드 추가
6. 부스터 추가 후 다시 단어 중복 정리

**검증**: M-314 5/5 모두 "명기"→"후방" 자동 교정 성공

**철학**: LLM은 80% 잘 하고 후처리가 나머지 20% 강제 보장. 둘 다 필요.

## D-014: 서버 구동 방식 — 단계별 전환

**상황**: FastAPI 서버를 어떻게 띄울 것인가

**고려한 후보**
- A) 명령어로 매번 띄우기 (`python src/server.py`)
- B) Windows 서비스로 등록
- C) 작업 스케줄러로 부팅 시 자동 시작

**선택**: **A로 개발 → C로 운영 전환**

**근거**
- A는 단순하고 로그가 콘솔에 바로 보여 개발·디버깅에 유리
- B는 설정 복잡 + 서비스 재시작 시 트러블슈팅 어려움
- C는 트레이 아이콘 안 보이지만 부팅 자동 시작 + 디버깅 시 수동 종료 후 재시작 가능
- 개발 끝나고 안정화 단계에 C로 전환하는 게 균형 잡힌 선택

**전환 조건**: 3·4단계까지 완성된 후 1주일 무사고 운영 시 C 적용

**검증**: 2단계 작업 내내 A 방식 운영, 2시간+ 무사고. C 전환은 7단계로 예정.

- B는 책임 분리가 깔끔해 보이지만, 결국 1단계 코드를 라이브러리화해서 의존성으로 묶어야 함

## D-016: 메인↔세컨 동기화 — SMB 네트워크 드라이브

**상황**: 메인 PC(개발)와 세컨 PC(실행) 간 코드 동기화 방식

**고려한 후보**
- A) SMB 네트워크 드라이브 매핑 (Z:\)
- B) Git 로컬 저장소 (push/pull)
- C) 파섹으로 세컨 PC 접속해 직접 코딩
- D) 수동 복사

**선택**: **A) SMB 네트워크 드라이브 매핑**

**근거**
- 메인 PC에서 안티그래비티로 Z:\ 폴더 열어 작업 → 저장하면 즉시 세컨 PC에 반영
- Git은 매번 push/pull 명령 부담
- 파섹은 사장님 메인 작업 환경(안티그래비티)을 못 살림
- 수동 복사는 잦은 수정 시 비효율

**구축 방법**
- 세컨 PC: `New-SmbShare -Name "renamer" -Path "C:\Users\pthem\Documents\adult_product_renamer" -FullAccess "yegom\pthem"`
- 메인 PC: `New-PSDrive -Name "Z" -PSProvider FileSystem -Root "\\192.168.0.200\renamer" -Credential "yegom\pthem" -Persist`

**검증**: 메인 PC에서 server.py 수정 → 세컨 PC에서 즉시 실행 가능 확인. robocopy로 1단계 코드 일괄 이동도 성공.

**주의**: venv는 PC별 환경 의존적이라 동기화하지 않음(robocopy에서 `/XD .venv` 제외). 세컨 PC에서 별도 venv 생성 필요.

## D-018: FastAPI의 진짜 부가 가치 — 모델 로딩 비용 제거

**상황**: 사장님 질문 "웹 형식이라 느려지는 거 아니냐?"

**답변 근거 (실측 기반)**

8초 응답의 정체:
```
전체 8초 중:
├── HTTP 통신 (메인↔세컨 2.5Gbps):  0.002초   ← FastAPI 오버헤드
├── Ollama 큐 대기:                   0.1초
└── qwen3:14b 실제 추론:              7.9초    ← 99.97%
```

**CLI vs FastAPI 비교**
| 방식 | 매 실행 시간 | 모델 로딩 비용 |
|---|---|---|
| 1단계 CLI (test_renamer.py) | 5~10초 첫 워밍업 + 추론 8초 = 13~18초 | 매번 발생 |
| 2단계 FastAPI | 서버 시작 시 1회 워밍업 → 이후 매 요청 8초 | 0 (메모리 상주) |

**결정**: FastAPI는 단순 API 노출이 목적이 아니라, **모델 로딩 비용 제거 + GUI 통합 준비**가 핵심 가치.

**검증**: 첫 호출 12.47초(워밍업 포함), 두 번째 호출부터 7.1초로 안정. uptime 2시간 동안 keep_alive 작동 확인.

## D-019: 회귀 테스트 의무화 + PowerShell 한글 깨짐 우회

**상황**: 2단계가 1단계와 동일한 결과를 내는지 검증 필요

**도입 도구**: `tests/test_api.py`
- 1단계 `test_renamer.py`와 같은 6개 카테고리 샘플 사용
- 차이는 직접 함수 호출 vs HTTP POST 호출만
- 출력 형식 완전 동일 → 결과 비교 쉬움

**환경 변수 지원**: `API_BASE_URL` 환경변수로 메인 PC(`http://192.168.0.200:8000`)와 세컨 PC(`http://localhost:8000`) 양쪽에서 실행 가능

**검증 결과**: 30/30 통과, 1단계와 동등 품질 확인

### 부산물: PowerShell `Invoke-RestMethod` 한글 깨짐 함정

**증상**: `irm`이나 `Invoke-RestMethod`로 호출 시 응답 한글이 깨짐

**원인**: 응답 바디를 자동 디코딩할 때 인코딩 추측 실패 → 이미 깨진 문자열이 변수에 저장됨. `$OutputEncoding`이나 `[Console]::OutputEncoding` 변경해도 **이미 깨진 데이터는 복구 불가**.

**해결 방법 3종**
1. **Swagger UI 사용** (브라우저, 가장 깔끔)
2. `Invoke-WebRequest` + `[System.Text.Encoding]::UTF8.GetString($response.RawContentStream.ToArray())` 수동 디코딩
3. Python 스크립트(`test_api.py`)에서 `sys.stdout.reconfigure(encoding='utf-8')` 사용

**채택 정책**: 운영 시 PowerShell 직접 호출은 권장 안 함. 개발 시엔 Swagger UI, 자동화 시엔 Python 사용.
