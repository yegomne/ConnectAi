# CASE STUDY — "_03 응답 없음" 멈춤 디버깅 (fit 진동 / 렌더 폭주)

> 문서 성격: 트러블슈팅 케이스 스터디 (지식화·구조화용).
> 특정 이미지(C-892-860_03)에서 프로그램이 "응답 없음"으로 멈추던 문제를
> 10+ 라운드에 걸쳐 진단·해결한 전 과정. 진범, 배제 과정, 진단 방법론, 교훈.
> 판매 제품이라 "어떤 이미지·GPU에서도 안 멈춰야" 했기에 끝까지 추적함.
>
> 작성: 2026-06-02 / 세현 사장님 / 항상 존댓말
> 관련: PHASE4-5c2(검수 타임아웃), PHASE4-5c3(멈춤 재진단)

---

## 0. 한 줄 결론

**진범 = fit 진동(무한 렌더 루프).** 세로로 긴 이미지를 화면에 맞추는(fitInView)
과정에서 스크롤바가 나타났다 사라졌다 하며 뷰 크기가 계속 바뀌어 fit→repaint가
무한 반복 → 메인 스레드 CPU 1코어 100% → "응답 없음".
**해결 = 스크롤바 AlwaysOn 고정 + fit 진동 가드** (paints가 4에서 안정 = 증거).
GPU 무관(순수 Qt 레이아웃 로직)이라 모든 환경에서 안전.

---

## 1. 증상

- 특정 이미지(C-892-860_03: 부드러운 살색 그라데이션이 넓은 상세컷)에서만 발생.
- 새 제품 [만들기] 시(=그 이미지 로드/표시 시) "응답 없음", 다른 버튼도 안 됨.
- 화면 하얗게 흐려짐(윈도우의 '응답 없음' 처리).
- 작업관리자: 메인 프로세스 CPU 8%(=1스레드 100%, Ryzen 5 7600 12스레드 기준), 16초.
- 다른 이미지(더 긴 L-828 750×3208 포함)는 정상. _03만.

---

## 2. 배제 과정 (용의자를 하나씩 제거)

순서대로 의심하고, 증거로 배제했다. **이 과정 자체가 방법론.**

| 용의자 | 어떻게 배제했나 | 결과 |
|---|---|---|
| 이미지 크기/길이 | L-828(비율 4.28)이 _03(1.75)보다 긴데 정상 | ✗ |
| 파일 손상 | Pillow로 직접 검사: 정상 RGB, 로드 0.04초 | ✗ |
| DecompressionBomb | _03=1.3MP, Pillow 제한 89MP의 1.4% | ✗ |
| 누끼(BiRefNet) | "누끼 사용" 꺼도 멈춤 | ✗ |
| 검수(NudeNet) | DISABLE_SCREENING=True(검수 OFF)해도 멈춤 + 워커 CPU 0% | ✗ |
| 저장 로직 | [만들기]만 했는데 멈춤(저장 전). [저장-진입0] 안 찍힘 | ✗ |
| 파이썬 코드 경로 | STEP A~D, 검수완료, tick 다 정상 통과 | ✗ |
| 포토샵 메타데이터 | clean 버전(adobe/xmp 제거)도 멈춤 | ✗ |
| 재인코딩/포맷 | 재인코딩=True여도 멈춤 | ✗ |
| 다운스케일 부족 | 629×1100으로 줄여도 멈춤 | ✗ |
| **fit 진동(렌더 루프)** | **스크롤바 AlwaysOn + fit 가드 → 해결, paints=4 안정** | **✓ 진범** |

핵심: 그림 픽셀 내용만 안 바뀐 채로 메타/크기/포맷을 다 바꿔도 멈춤 →
"이미지를 화면에 그리는 렌더링"으로 좁혀짐.

## 3. 결정적 진단 도구 (이번에 효과 본 것들)

진단을 가능케 한 로그/도구. **다음에 비슷한 문제 시 똑같이 쓴다.**

1. **ALIVE tick 로그**: 1~2초마다 "프로그램 살아있음 + 상태" 출력.
   - tick이 계속 찍히면 = 파이썬 안 멈춤(렌더/외부 문제).
   - tick 간격이 벌어지면(예: 16초) = 그 사이 메인 블록.
2. **paints 카운터**: paintEvent 호출 횟수를 tick에 같이 출력.
   - paints가 수백씩 폭증 → 무한 렌더 루프(fit 진동) 확정.
   - paints=4에서 안정 → 루프 멈춤 = 해결 증거. **이게 진범 확정의 결정타.**
3. **단계별 STEP 로그**: load/검수/저장/표시를 한 줄씩 print(flush=True).
   - 멈추기 직전 마지막 로그 = 멈춘 지점.
   - "검수완료 — 이후 멈추면 렌더/외부 원인" 같은 경계 로그가 특히 유용.
4. **작업관리자(CPU/프로세스)**: 어느 python이 CPU 먹는지.
   - 워커 0% + 메인 8% → 검수 아니라 메인 렌더링 확정.
5. **진단 플래그 DISABLE_SCREENING**: 검수를 코드로 완전히 꺼서 검수 배제.
6. **Pillow로 파일 직접 검사**(크기/모드/메가픽셀/메타데이터): 파일 무죄 확인.

## 4. 진범 메커니즘 (fit 진동)

```
이미지 로드 → fitInView(화면맞춤)
   → 세로 긴 이미지라 세로 스크롤바 생김
   → 뷰포트 가용 폭 줄어듦 → 크기 변함
   → resizeEvent → 다시 fitInView
   → 스크롤바 사라짐 → 폭 늘어남 → 또 fitInView
   → 스크롤바 생김 → … 무한 반복
   → paintEvent 폭주, CPU 1스레드 100%, 16초+ → 응답 없음
```
- _03이 특정 비율이라 "스크롤바 경계"에 딱 걸려 진동이 안 끝남.
- 다른 이미지는 그 경계에 안 걸려서 fit 1~2회로 안정.

## 5. 해결 (적용된 변경)

진범 해결(#5) + 범용 성능 보조. 안티그래비티가 누적 적용한 6개 중:

| # | 변경 | 역할 | 판정 |
|---|---|---|---|
| 1 | 표시용 다운스케일 | 화면 픽셀↓(좌표/저장은 원본) | 보조(성능) |
| 2 | 렌더힌트 OFF(AA/스무딩) | 렌더 부담↓ | 보조 |
| 3 | 픽스맵/뷰 캐시 + FastTransformation | 1회 렌더 후 재사용 | 보조 |
| 4 | 재인코딩(메타 제거) | EXIF/XMP/Adobe 제거 | 효과 없음(메타 무죄) |
| **5** | **스크롤바 AlwaysOn + fit 진동 가드** | **무한 렌더 루프 차단** | **★ 진범 해결** |
| 6 | AA_UseSoftwareOpenGL | RX570 드라이버 배제 | 이 앱선 거의 no-op |

- **진범 해결 = #5** (paints=4 안정이 증거).
- #1 다운스케일은 부작용 주의: '긴 변' 기준이면 세로 긴 이미지(예: 750×6527)가
  가로 126px로 뭉개져 작업 불가 → **가로 기준 또는 임계 크게로 수정**(아래 6).
- #4(재인코딩), #6(소프트웨어 OpenGL)은 진범 아님 → 정리 대상(하나씩 빼며 재검증).

## 6. 후속 부작용 + 수정 (다운스케일)

- 증상: 세로로 매우 긴 상세페이지(C-682 750×6527)가 화면에서 흐릿/뭉개짐 → 작업 불가.
- 원인: 표시용 다운스케일이 '긴 변 1100' 기준 → 가로가 126px로 축소.
- 수정: 다운스케일을 **가로 기준**(또는 임계 크게)으로 → 가로 선명도 유지.
  · 진범은 #5라 다운스케일 완화해도 _03 멈춤 재발 안 함(확인됨).
- 교훈: 세로 긴 이미지가 많은 도메인이라, 스케일 기준은 '가로'가 안전.

## 7. 마무리 정리 (디버깅 흔적 제거)

- cmd 콘솔 숨김: python → **pythonw**(콘솔 없이 GUI만). 판매 제품용 깔끔.
- 디버그 로그: **DEBUG 플래그로 묶기**(평소 False=안 찍힘, 문제 시 True로 재활용).
  · 완전 삭제 X — 이번처럼 엣지 케이스 재발 시 다시 써야 함.
- DISABLE_SCREENING: False(검수 정상)로.
- 진범 코드(#5 fit 가드, 가로기준 다운스케일)는 절대 건드리지 않음.

## 9. 교훈 (방법론 — 다음에 또 쓴다)

1. **"응답 없음"은 두 종류**: ①파이썬 코드 블록 ②렌더/외부.
   ALIVE tick으로 구분 — tick 계속 찍히면 코드 무죄, 렌더 의심.
2. **작업관리자 CPU가 진실**: 어느 프로세스가 CPU 먹는지로 워커/메인 가른다.
3. **용의자는 코드로 끄고 배제**: DISABLE_SCREENING처럼 의심 기능을 플래그로 꺼서
   배제하면 추측이 사실로 바뀐다.
4. **파일은 직접 까본다**: Pillow로 크기/모드/메가픽셀/메타 확인 → 파일 무죄 빠르게 확정.
5. **무한 루프는 카운터로 잡는다**: paintEvent 같은 의심 호출에 카운터 → 폭증이면 루프.
6. **사용자 증언 정밀화**: "저장에서 멈춤"으로 좁혔다가 실제론 "만들기에서"였음.
   정확한 재현 단계(클릭 순서)를 받는 게 결정적. 추측으로 엉뚱한 곳 고치면 헛수고.
7. **그림 내용만 고정하고 나머지 바꿔보기**: 메타/크기/포맷 다 바꿔도 멈추면
   "픽셀을 화면에 그리는 렌더링"으로 범위가 확 좁혀진다.
8. **판매 제품은 끝까지**: 엣지 케이스라도 "어떤 입력·환경에서도 안 죽게". 우회 금지.

## 10. 함정 추가 (RUNBOOK 보강, 39~)

39. QGraphicsView + fitInView + 자동 스크롤바 = 특정 비율 이미지에서 fit 진동
    (무한 렌더 루프) → 응답없음. 스크롤바 AlwaysOn 고정 + fit 가드로 차단.
40. "응답없음" 진단: ALIVE tick + paints 카운터 + 작업관리자 CPU 3종으로
    코드/렌더/워커 중 어디인지 가른다.
41. 표시용 다운스케일은 '가로 기준'. '긴 변' 기준이면 세로 긴 이미지 뭉개져 작업 불가.
42. 진단 후 디버그 로그는 지우지 말고 DEBUG 플래그로 보존(재발 대비). 실행은 pythonw.
43. 사용자 증언("저장에서 멈춤")을 그대로 믿지 말고 정확한 재현 단계 확보.
    실제 멈춤 지점은 "만들기(이미지 로드)"였음.
